# `ADR` Routing API refactor

The gate API introduced in [the previous refactor](./2026-03-09.route-api-refactor.md) conflates two distinct concerns: URL structure and whether a URL is a valid destination. These are separate questions — but `junction` encodes both at once. A `junction` gate declares its URL pattern and simultaneously declares that the URL is not a valid destination. That coupling makes the concept harder to explain and forces a binary choice at definition time that may need to change later without touching the gate itself.

Beyond `junction`, the implicit parent-child rendering stack is opaque. When a junction is active, it renders its component and auto-activates its child gate — behavior that is invisible at the component level and only discoverable by knowing the junction semantics. The `hasParent` implementation that drives this adds complexity to the router without a clear boundary of responsibility.

A further issue with the previous draft (routing API refactor) is that it placed URL pattern definition in `gate` and destination declaration in `route()`. This has the primary primitive backwards: routes describe the application's navigational structure, gates are a rendering detail that subscribes to them.

## Decision:

Refactor the routing API to three concrete changes:

**1. `route` becomes the tagged template literal for URL patterns**

```ts
// Before — URL patterns are defined on gates
export const CategoriesGate = gate`/categories/`(Categories)
export const CategoryGate   = gate`/${CategoriesGate}/${token('slug', String)}/`(Category)

// After — URL patterns and destinations are defined on routes
export const CategoriesRoute = route`/categories/`(Categories)
export const CategoryRoute   = route`/${CategoriesRoute}/${token('slug', String)}/`(Category)
```

`route` is a tagged template literal that takes a URL pattern and binds a component to it as the top-level destination component. Routes are the primary definition of the application's navigational structure. Parent routes can be interpolated into child route patterns to compose URL paths, in the same way gates previously used parent interpolation.

**2. `gate(RouteRef, Component)` subscribes a component to a route**

```ts
// Before — gate defines the URL and implicitly owns rendering
export const CategoryGate = gate`/${CategoriesGate}/${token('slug', String)}/`(Category)

// After — gate subscribes to an existing route
export const CategoryGate = gate(CategoryRoute, Category)
```

`gate` is now a plain function call, not a tagged template. It takes a route reference and a component, and returns an embeddable unit that activates when the route's URL pattern matches. Gates are what components pass to `append()` to conditionally render sub-content; they subscribe to a route rather than defining one. A route can exist without a gate — it is still a navigational destination, just not embeddable as a sub-component.

**3. `junction` and `hasParent` removed**

`junction` is eliminated entirely. Routes always declare navigational destinations; there is no concept of a URL pattern that is not a destination. If a section of the app has a URL prefix used only for child composition, it is defined as a route with its own destination component — the distinction between "real destinations" and "pass-through junctions" does not need to exist in the API.

The `hasParent` flag and its associated router logic are removed. Parent interpolation in route patterns (`` route`/${ParentRoute}/segment/` ``) resolves the child's full URL pattern at definition time and is used only for URL composition. The router no longer needs to distinguish root routes from child routes.

### Drivers

#### Routes are the primary primitive

A routing library's foundational concept is the route. Having `gate` own the URL pattern — with `route` as a secondary grouping construct — inverts the natural hierarchy. `route` should declare what the application's destinations are; `gate` should be a rendering detail that says "show this component when this route is active." The name reflects the purpose: a gate opens or closes based on whether its subscribed route matches.

#### URL composition belongs on routes

Parent interpolation (`` route`/${ParentRoute}/segment/` ``) reads as "this route extends that route's path," which is a natural description of URL structure. With gates owning the pattern in the previous model, the same interpolation carried rendering implications via `hasParent`. Moving it to `route` makes it purely compositional.

#### Inclusion is self-evident; `junction` requires explanation

The `junction` semantics — "renders only when a child is active, otherwise falls through" — are non-obvious and require documentation. Every route being a destination is the simpler default: navigating to a URL either matches a route or it does not. The need for non-destination URL prefixes is rare enough that it does not warrant a named concept.

#### Route components own their rendering

A route component that uses `append(CategoryGate, {})` explicitly declares that it shows category content when the category route is active. This is visible, flexible, and does not depend on any implicit parent-child activation chain. The component decides what to render; the router only decides which route matches.

#### Sub-routing is an advanced, opt-in pattern

Parent route interpolation enables additive URL composition for cases where it is genuinely needed. This is an advanced pattern that most frameworks do not provide at all; it should not be the default mode of thinking about routes. The majority of routes should be independent declarations with no parent interpolation. Sub-routing is available when the URL hierarchy demands it, but rendering the chain is always the route component's explicit choice.

### Alternatives

#### Previous draft — `gate` as template, `route()` as destination grouper

```ts
export const CategoriesGate  = gate`/categories/`(Categories)
export const CategoryGate    = gate`/${CategoriesGate}/slug/`(Category)
export const CategoriesRoute = route(Categories, CategoryGate)
```

Rejected. This model placed URL structure on `gate` and destination policy on `route()`, which left `gate` as the primary primitive. The composition was also awkward: the component appeared on both the gate and the route call, and the route listed gates rather than defining structure directly.

#### Destination flag on `router()` — passing terminal gates separately

```ts
router({ home, notFound, ...appRoutes }, { terminal: [CategoryGate] })
```

Rejected. This separates the destination declaration from both the pattern definition and the component that owns the section. It puts routing policy into the top-level router call, which is already concerned with registration.

#### Keep `junction`, add `route` alongside

Allow `junction` to coexist with the new `route` primitive so teams can migrate incrementally. Rejected. Having two mechanisms for destination control creates ambiguity. The `junction` concept is the problem being solved, not something to preserve.

## Outcome

A categories section where `/categories/` is its own destination and child routes are embeddable via gates:

```ts
// _gates.mts
import { route, token, gate } from '@rooted/router'
import { Categories } from './categories.mjs'
import { Category }   from './category.mjs'
import { Recipe }     from '../recipes/recipe.mjs'

export const CategoriesRoute = route`/categories/`(Categories)
export const CategoryRoute   = route`/${CategoriesRoute}/${token('slug', String)}/`(Category)
export const RecipeRoute     = route`/${CategoryRoute}/recipes/${token('id', Number)}/`(Recipe)

export const CategoryGate = gate(CategoryRoute, Category)
export const RecipeGate   = gate(RecipeRoute, Recipe)
```

```ts
// categories.mts — the route component appends child gates explicitly
export const Categories = component({
  name: 'categories-page',
  onMount({ append, create }) {
    append('div', { className: 'category-grid', children: mapCategories(create) })
    // Visible when CategoryRoute is active
    append(CategoryGate, {})
    // Visible when RecipeRoute is active
    append(RecipeGate, {})
  },
})
```

Navigating directly to `/categories/italian/recipes/1/` matches `RecipeRoute`. The `Categories` component renders (it is the destination component for `CategoriesRoute`, which is the URL parent) and mounts both `CategoryGate` and `RecipeGate`. Each gate self-manages its own visibility against the current URL, so deep-linking works without any additional coordination.

### Pros and cons

`pro` `route` is the primary primitive — names match their purpose \
`pro` `junction` eliminated — every route is a destination, no concept to learn \
`pro` `hasParent` removed — router implementation is simpler \
`pro` Gate subscriptions are explicit — `gate(RouteRef, Component)` reads as a clear binding \
`pro` Route components own their rendering — no implicit auto-activation of child gates \
`con` Breaking change — all gate definitions must be rewritten as route + gate pairs \
`con` Child gate composition is now explicit — `append` calls must be added to route components \
`con` The same component often appears in both the `route` and the `gate` call for the common case
