# `ADR` Routing API refactor

The gate API introduced in [the previous refactor](./2026-03-09.route-api-refactor.md) conflates two distinct concerns: URL structure and whether a URL is a valid destination. These are separate questions — but `junction` encodes both at once. A `junction` gate declares its URL pattern and simultaneously declares that the URL is not a valid destination. That coupling makes the concept harder to explain and forces a binary choice at definition time that may need to change later without touching the gate itself.

Beyond `junction`, the implicit parent-child rendering stack is opaque. When a junction is active, it renders its component and auto-activates its child gate — behavior that is invisible at the component level and only discoverable by knowing the junction semantics. The `hasParent` implementation that drives this adds complexity to the router without a clear boundary of responsibility.

## Decision:

Refactor the routing API to three concrete changes:

**1. `route()` as the explicit destination declaration**

```ts
// Before — destination eligibility is a property of the gate type
export const CategoriesGate = junction`/categories/`(Categories)
export const CategoryGate   = gate`/${CategoriesGate}/${token('slug', String)}/`(Category)

// After — destination eligibility is declared separately
export const CategoriesGate = gate`/categories/`(Categories)
export const CategoryGate   = gate`/${CategoriesGate}/${token('slug', String)}/`(Category)

export const CategoriesRoute = route(Categories, CategoriesGate, CategoryGate)
```

`route(Component, ...gates)` returns a `RouteMap` — a flat map binding each gate's resolved URL pattern to the given component. The router accepts a `RouteMap` directly (via spread). All listed gates activate the same top-level component; which sub-content renders is the component's own responsibility.

To exclude a gate as a destination, omit it from any `route()` call:

```ts
// CategoriesGate URL exists for child composition but is not a navigational destination
export const CategoriesRoute = route(Categories, CategoryGate)
```

**2. `junction` removed**

`junction` is eliminated entirely. Its two jobs are now handled separately: URL composition remains in the gate's parent interpolation, and destination eligibility is controlled by `route()` membership. There is no longer a gate type that "falls through to `notFound`" — that behavior is simply the absence of a matching `route()` entry.

**3. `hasParent` removed**

The `hasParent` flag and its associated logic in the router are removed. Parent interpolation (`gate\`/${ParentGate}/segment/\``) continues to work for URL composition — it resolves the child's full pattern at definition time — but it no longer implies a rendering relationship. The router no longer needs to distinguish root gates from child gates.

### Drivers

#### Destination eligibility should be a declaration, not a gate property

Whether a URL is a valid destination can change independently of what pattern it matches or what component it mounts. Encoding it in the gate type (`gate` vs `junction`) means changing destination eligibility requires changing the gate definition — which may be in a different file, owned by a different concern. A `route()` call in the same file as the component that owns the section makes the declaration local to the right place.

#### Inclusion is self-evident; `junction` requires explanation

The `junction` semantics — "renders only when a child is active, otherwise falls through" — are non-obvious. Every new developer encountering a `junction` must learn what it means. Destination-by-exclusion needs no explanation: if a gate is not in a `route()`, navigating to its URL simply does not match.

#### Route components own their rendering

The implicit rendering stack (junction activates child automatically) ties rendering behavior to the URL hierarchy. A route component declaring its own `append(ChildGate, {})` is explicit, visible at the component level, and flexible — the component can choose different rendering strategies for different paths without any router involvement.

#### Sub-routing is an advanced, opt-in pattern

Parent gate interpolation enables additive URL composition for cases where it is genuinely needed. This is an advanced pattern that most frameworks do not provide at all; it should not be the default mode of thinking about routes. The majority of routes should be independent `route()` declarations. Sub-routing via gate interpolation is available when the URL hierarchy demands it, but rendering the chain is always the route component's explicit choice.

### Alternatives

#### Destination flag on `router()` — passing terminal gates separately

```ts
router({ home, notFound, ...appRoutes }, { terminal: [CategoryGate] })
```

Rejected. This separates the destination declaration from both the gate definition and the component that owns the section. It also puts routing policy into the top-level router call, which is already concerned with registration, not destination policy.

#### Keep `junction`, add `route()` alongside

Allow both `junction` and `route()` to coexist so teams can migrate incrementally. Rejected. Having two mechanisms for controlling destination eligibility creates ambiguity about which to use and in what combination. The value of `route()` is that it is the single place where destinations are declared; `junction` undermines that.

#### Auto-derive routes from gate hierarchy

Generate a `RouteMap` automatically from the gate tree, treating all leaf gates as valid destinations and all non-leaf gates as pass-through. Rejected. "Leaf gate" is not a stable concept — a gate can gain or lose children as the application evolves. Implicit destination eligibility based on structural position is harder to reason about than an explicit `route()` list.

## Outcome

A categories section with one non-routable parent gate and two routable child gates:

```ts
// _gates.mts
import { gate, token, route } from '@rooted/router'
import { Categories } from './categories.mjs'
import { Category }   from './category.mjs'
import { Recipe }     from '../recipes/recipe.mjs'

export const CategoriesGate = gate`/categories/`(Categories)
export const CategoryGate   = gate`/${CategoriesGate}/${token('slug', String)}/`(Category)
export const RecipeGate     = gate`/${CategoryGate}/recipes/${token('id', Number)}/`(Recipe)

// /categories/ is not a destination; only /categories/:slug/ and deeper are routable
export const CategoriesRoute = route(Categories, CategoryGate, RecipeGate)
```

```ts
// categories.mts — the route component appends child gates explicitly
export const Categories = component({
  name: 'categories-page',
  onMount({ append, create }) {
    append('div', { className: 'category-grid', children: mapCategories(create) })
    // Visible when path matches /${CategoriesGate}/${token('slug', String)}/
    append(CategoryGate, {})
    // Visible when path matches /${CategoryGate}/recipes/${token('id', Number)}/
    append(RecipeGate, {})
  },
})
```

Navigating directly to `/categories/italian/recipes/1/` renders `Categories`, which mounts both `CategoryGate` and `RecipeGate`. Each gate self-manages its own visibility against the current URL, so deep-linking works without any additional coordination.

### Pros and cons

`pro` Destination eligibility is an explicit, readable declaration — no concept to learn \
`pro` `junction` eliminated — one fewer named export and one fewer concept \
`pro` `hasParent` removed — router implementation is simpler \
`pro` Route component owns rendering — no implicit auto-activation of child gates \
`pro` Changing which URLs are valid destinations does not require touching gate definitions \
`con` Breaking change — all `junction` definitions must be converted \
`con` Child gate composition is now explicit — `append` calls must be added to route components \
`con` `route()` is a new concept alongside `gate()`
