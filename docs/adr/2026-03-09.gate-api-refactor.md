# `ADR` Gate API refactor

```yml

created: 2026-03-09
deciders:

```

The current gate API has two friction points. First, argument order is inverted relative to how developers think about routes â€” the component is declared before the URL pattern, but the URL is the primary concern when defining a route. Second, the `.exact` modifier is misleading: it sounds like "match this path exactly" but actually means the opposite â€” the gate renders only when the path has *more* after it. The name is a source of confusion and the modifier-chaining style (`gate(Component).exact`) is inconsistent with a cleaner path-first approach.

Beyond naming, the `.append()` child-route composition pattern is awkward: the component and the path segment are passed at different call sites, making the child definition harder to read and type.

## Decision:

Refactor the gate API to four concrete changes:

**1. Path-first argument order**

```ts
// Before
export const ArticleGate = gate(Article)`/articles/${token('id', Number)}/`

// After
export const ArticleGate = gate`/articles/${token('id', Number)}/`(Article)
```

The URL pattern becomes the primary declaration; the component is what gets bound to it.

**2. Parent interpolation replaces `.append()`**

```ts
// Before
export const CommentsGate = ArticleGate.append(Comments)`/comments/`

// After
export const CommentsGate = gate`${ArticleGate}/comments/`(Comments)
```

A parent gate interpolated into a child's template string replaces the `.append()` method. The relationship between parent and child is visible at the child's definition site. The runtime detects that the interpolated value is a `BoundGateDefinition` (via the `typedParameter` symbol) and uses it as the parent reference. Parent params are merged into child params as before.

**3. `junction` replaces `.exact`**

```ts
// Before
export const ArticleGate = gate(Article).exact`/articles/${token('id', Number)}/`

// After
export const ArticleGate = junction`/articles/${token('id', Number)}/`(Article)
```

`junction` is a separate named export. A junction renders its component only when a child route is also active; navigating to the junction's own path falls through to `notFound`. This is the same behaviour as `.exact` but expressed as a first-class concept rather than a modifier. The term was chosen because a junction is a routing point that leads elsewhere â€” it has a parent relationship to its child gates, and carries a control-flow connotation without being associated with concurrency primitives.

**4. `wildcard` interpolation for catch-all matching**

```ts
import { gate, token, wildcard } from '@rooted/router'

export const ArticleGate = gate`/articles/${token('id', Number)}/${wildcard}/`(Article)
```

A `wildcard` interpolation in the URL pattern matches one or more path segments. Unlike a junction, a gate with a wildcard *does* render at its own prefix path â€” the wildcard simply broadens what it matches. This is a new capability with no equivalent in the current API.

### Drivers

#### Path-first reads more naturally

When reading a route definition, the URL tells you *where* and the component tells you *what renders there*. Having the component come first inverts that. Path-first matches how every file-based router and documentation example describes routes.

#### Template interpolation is a natural composition primitive

Tagged template literals already accept interpolations as first-class values via `token()`. Extending that to accept a parent `BoundGateDefinition` follows the same pattern â€” the template string describes the full URL, parameters and all, and the runtime distinguishes token interpolations from gate interpolations by type.

#### `junction` names a real concept

The distinction between a gate (a destination) and a junction (a routing point that leads to child destinations) is fundamental. It appears in every layered router: layout routes vs page routes in file-based systems, exact vs prefix routes in imperative systems. Surfacing it as a named import rather than a modifier makes it discoverable, self-documenting, and impossible to accidentally omit.

#### `wildcard` belongs in the URL pattern

A wildcard is a structural property of the URL pattern, not a behavioral flag. Placing it as an interpolation in the template string keeps it alongside `token()` where URL structure is already declared.

### Alternatives

#### Option A â€” children as named object map

```ts
export const ArticleGate = gate`/articles/${token('id', Number)}/`(Article, {
  Comments: gate`/comments/`(Comments)
})
```

Rejected because child gates become implicit named properties on the parent (`ArticleGate.Comments`) rather than independent named exports. TypeScript typing for the dynamic child map is complex, and the nesting makes it harder to read and to discover individual child gates.

#### `.exact` modifier kept, argument order flipped

```ts
export const ArticleGate = gate`/articles/${token('id', Number)}/`.exact(Article)
```

Rejected. Moving argument order was straightforward; `.exact` remained misleading regardless of where it appeared. The modifier style also splits the URL pattern from the behavioral intent across two call sites.

#### `layout` as named export

```ts
export const ArticleGate = layout`/articles/${token('id', Number)}/`(Article)
```

Rejected. A junction renders a real component with meaningful content, not a structural wrapper. Calling it a `layout` implies it exists only to wrap children, which is incorrect.

#### `required` as second argument sentinel

```ts
export const ArticleGate = gate`/articles/${token('id', Number)}/`(Article, required)
```

Rejected. The second argument is easy to overlook, and `required` is ambiguous â€” it reads as a validation constraint rather than a routing concept. The distinction between a gate and a junction is fundamental enough to deserve its own named export.

#### Alternative names for `junction`

Considered in order: `branch` (too vague, no routing connotation), `gateway` (strong fit but too close to API gateway in web infrastructure), `parent` (accurate but purely relational, no control-flow sense), `fork` (better than branch but still metaphorical), `hub`, `scope`, `stem`. `junction` was selected because it carries a switching/routing connotation (rail junctions, road junctions) while staying close to the parent-child relationship and avoiding association with threading primitives like semaphore.

#### In-pattern sentinel for junction behaviour

```ts
export const ArticleGate = gate`/articles/${token('id', Number)}/${subRoutes}/`(Article)
```

Rejected. `subRoutes` (or any similar sentinel) is a routing directive embedded in what should be a URL description. `token()` belongs in the template because it describes a URL segment; a "requires child" sentinel does not describe any segment â€” it describes behavior. Mixing the two concerns into the template is why `subRoutes` felt wrong regardless of its name.

## Outcome

The refactored API for a parent-child route pair:

```ts
import { gate, junction, token, wildcard } from '@rooted/router'

// Junction: renders Article only when a child route is active
export const ArticleGate    = junction`/articles/${token('id', Number)}/`(Article)
// Gate: renders Comments at /articles/123/comments/
export const CommentsGate   = gate`${ArticleGate}/comments/`(Comments)
// Gate with wildcard: renders Article for any path under /archive/
export const ArchiveGate    = gate`/archive/${wildcard}/`(Archive)
```

### Pros and cons

`pro` URL pattern is the primary declaration â€” argument order matches the mental model \
`pro` Parent-child relationship is visible at the child definition site \
`pro` `junction` is a first-class, named, discoverable concept \
`pro` `wildcard` adds catch-all capability with no new syntax beyond what `token()` already established \
`pro` No modifier chaining â€” each gate type is a single import \
`con` Breaking change to all existing gate definitions \
`con` `.append()` removal requires updating all child gate definitions \
`con` Two named imports (`gate`, `junction`) where one (`gate`) existed before
